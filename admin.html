DOCS — Top Banner

1) What this is (in one sentence)

CFAMM is a signup-free link shortener + browsing flow that earns ad revenue, then (optionally) shares a fixed portion via a transparent pool model.

Share link → people browse → ads earn → revenue settles → pool is funded → payout is proportional.

2) The only rule that matters

No revenue = no payout.
Ads must earn money and the ad network must pay out for the month before any pool distribution can exist. Anything else is fantasy.

3) How payouts actually work (simple math)

Pool = min( monthly_cap , received_revenue_this_month × pool_percent )
Your share = your_units ÷ total_units
Your payout = pool × your_share

Units are NOT dollars. Units are just a contribution counter used to compute percentages. The pool is funded after revenue exists.

4) What’s real vs. what’s local (important)

Local (this browser only):
  • Stats, projections, sponsor schedules, monthly summaries
  • Admin settings (PIN unlock), density toggles, context skins
Real-world (requires deployment + ad network):
  • Actual ad revenue
  • Actual settlement totals
  • Actual sponsor payments
This file is intentionally designed so you can test flow and UX safely without pretending revenue is real.

5) Policy safety (how we avoid bans)

This build is designed to stay within ad-network rules:
  • No auto-refresh timers for ads
  • No incentivized clicks (no “click to earn”, no “watch to earn”)
  • Ads refresh only on user intent (navigation / step change)
  • Support modal is optional and not tied to rewards
If you break these rules, you risk losing the ad account. Don’t.

6) Why this can earn even at low traffic (10–250 DAU)

Most sites die at low traffic because they have low intent and low session depth. CFAMM focuses on:
  • High-intent framing (calculator / estimator / reference contexts)
  • Session depth via legitimate “next step” routes
  • High viewability (anchor + side inventory stays visible)
  • Sponsors that buy presence, not volume
Low traffic won’t make you rich — but it can still be worth running.

7) Context Skins (how to raise CPM without changing layout)

Context Skins change title/meta/copy so different advertiser auctions compete. Same layout. Same ads. Better bids.
  • Calculator: utility intent
  • Estimator: decision intent
  • Reference: authority intent
  • Checklist / 2026: urgency/recency signals
Tip: clones across multiple domains/paths compound this effect.

8) Geo Tier Density (why we hide ads for low-value traffic)

Some traffic is low-CPM. Overserving it hurts RPM overall. Geo Tier Density lets you render fewer placements for Tier-3 while keeping Tier-1 premium.
  • Tier‑1: full inventory
  • Tier‑2: balanced
  • Tier‑3: light (protects RPM)

9) Sponsored Stickies (the low-DAU income hack)

Sponsors buy 72 hours of visibility — not clicks and not promises. This works even at low DAU because it’s “presence” inventory.
Suggested tiers (same layout, different coverage):
  • Base: one sticky slot
  • Plus: sticky + one additional placement
  • Premium: sticky + all contexts (Calculator/Estimator/Reference)
Always keep language neutral: “placement / visibility”, never “guaranteed results”.

10) Demand stacking + header bidding (when you’re ready)

This file is ready for demand stacking, but it is not included by default.
  • Start with one ad network until stable traffic
  • Add a fallback network if fill is weak
  • Only consider managed header bidding after you have steady usage and clean policy history
Header bidding can lift RPM, but it adds complexity. Don’t rush it.

11) Quick start checklist (deploy)

  1. Host the file on static hosting (anywhere)
  2. Replace placeholder ad blocks with your network tags
  3. Keep no timers, no incentives, navigation only
  4. Use Admin to tune density + context skins
  5. Track outcomes with real analytics (optional)

12) FAQ (common confusion)

Q: Do I need an account?
A: No. This build is signup-free by design.

Q: Are payouts guaranteed?
A: No. Payouts exist only after real revenue settles.

Q: Can I refresh ads automatically?
A: Don’t. It’s one of the fastest ways to get banned.

Q: What matters most at low traffic?
A: Session depth, viewability, context quality, and sponsors.

Ad Funnel & Processing (How revenue happens)

This page explains the ad flow in plain language. This is a static single-file build: it shows placeholders and logs user-intent navigation. Real ad revenue is created only when you publish with a real ad provider and follow their policies.

1) The funnel (why the site earns)

  • Traffic enters via a share link (UTMs / referrer) or direct visit.
  • Users navigate by intent through a short step flow (no timed refresh). Each step is a separate view event.
  • Each view exposes ad inventory (banner/rail/in-content + sticky anchor). More legitimate views = more legitimate impressions.
  • Stats are logged locally (pageviews, sessions, step reach, device share, ads shown). These help you optimize PV/session.

2) Ad inventory on the site

  • Top Banner — primary above-the-fold placement.
  • Desktop Rail — visible on larger screens (hidden on mobile).
  • In‑Content (Tall) — mid/late scroll placement.
  • Sticky Anchor (bottom) — persistent, always-visible (policy-safe when not outcome-linked and not forcibly gated).
  • Side Stickies (optional) — left + right sponsored slots. Hideable by the user; reappear on refresh.

3) What this build logs vs what an ad network pays

  • This build logs: view counts, session count, step flow reach, estimated “ads shown”, top UTM, device share.
  • Ad networks pay: based on measurable impressions / viewability / fill rate / geography / advertiser demand.
  • Important: this app never claims guaranteed earnings. It shows planning math and transparent counters.

4) Policy safety (why we avoid refresh spam)

  • No auto-refresh timers. Impressions happen through user intent (navigation/steps).
  • No “watch ads to win” language. Ads are not tied to outcomes.
  • Optional hide controls exist for side placements (user control).
  • Neutral wording: “sponsored” and “disclosed odds / demo mode” style language only.

72‑Hour Sponsored Stickies (Paid Rotation)

Sponsors are scheduled in 72 hour windows. A window can contain up to 9 sponsors. This UI can display up to 3 at a time (bottom + left + right), rotating by user navigation (not timers).

How sponsors are selected

  • Only sponsors within their active 72h window are eligible.
  • We pick up to 3 unique active sponsors per view cluster.
  • Rotation changes on user intent (page/step changes), never on time.

If no sponsors are booked

  • The side stickies fall back to normal ad inventory labels (Desktop Rail / In‑Content Tall) so premium positions still show revenue‑earning placements.
  • The permanent bottom sticky remains available as the anchor placement.

Sponsor quality rules

  • No malware, scams, or deceptive links.
  • No promise-based wording (“guaranteed payout”, “risk-free”, etc.).
  • Sponsors can be removed if abusive or policy-risky.

ADMENSION Pool Math (Transparent, post‑revenue)

  • Users earn participation units during the month (not dollars).
  • After the month closes and ad networks actually pay, you enter the received revenue for settlement.
  • 33% of net received ad revenue is allocated to the ADMENSION pool (cap applies).
  • User payout is proportional: pool × (your_units / total_units).
  • Minimum redeem threshold can be enforced (e.g., $20) and payouts are only from money already received.

RPM Optimization Manual (Ad-only, policy-safe)

This system is designed to maximize legitimate ad revenue via user-intent navigation and high viewability. It does not use auto-refresh timers or incentivized ad interaction.

1) What this site is

  • A step-based pageflow that increases pages per session and time-on-site.
  • Persistent stickies keep ads viewable for longer without popups.
  • Impressions are driven by navigation and step progression — not refresh spam.

2) Placements (defined once)

  • Bottom Sticky Anchor — always visible, stable baseline inventory.
  • Side Stickies — persistent on desktop, hideable by the user (state remembered).
  • Top Banner — above-the-fold, shown on main routes.
  • In-Content Tall — appears after engagement/steps for higher viewability.
  • Desktop Rail — desktop-only premium rail inventory.

3) User-intent impressions (why this is compliant)

  • No timed refreshes or forced reloads.
  • Ad exposure increases only when a user chooses to proceed (route/step).
  • We log ad-intent counts for transparency; real revenue is reported by ad networks.

4) Progressive reveal (viewability-first)

  • Bottom sticky is immediate.
  • Side stickies appear after first interaction.
  • In-content placements appear after step progression.
  • This improves viewability and reduces banner blindness.

5) Geo-weighted density

Higher-paying geographies can support more placements without harming RPM. Lower CPM traffic uses a lighter density to keep average RPM higher.
  • Tier-1: full inventory
  • Tier-2: bottom + side + core in-content
  • Tier-3: bottom + minimal (no side)

6) What we explicitly do NOT do

  • No auto-refresh timers
  • No “click ads to continue” gating
  • No incentivized clicks
  • No fake countdowns tied to ads

7) Stats vs payouts

  • Stats page shows logged sessions/PVs/ad-intent for tuning.
  • Ad networks determine actual revenue (CPM, fill rate, viewability).
  • ADMENSION pool settlement is computed only after revenue is received.

Advanced Publisher Stack (v1.2)

This build adds publisher-grade knobs and diagnostics: funnel conversion, geo-tier density tuning, context skins, duration bins, and placement-weighted RPM estimation — while staying policy-safe (no refresh timers, no incentivized clicks).

Decision rules (stair-climb)

  • If Step 1→2 is < 55%: improve copy/UX before adding inventory.
  • If pages/session is < 2.5: add more optional “Explore” routes (no timers).
  • If Tier-1 share is low: prioritize Tier-1 distribution (CA/US/UK/AU) before tuning CPM.
  • If RPM is flat while sessions rise: add demand (waterfall/header bidding), not more steps.
  • When one domain stabilizes: clone this engine into new context skins on new domains.

Compliance stance

  • No auto-refresh timers.
  • No “click ads to continue” gating.
  • No incentivized clicking.
  • Users can hide side stickies and the state is remembered.
